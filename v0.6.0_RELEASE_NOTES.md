## 1. What v0.6.0 delivered

- Replaced the Create React App toolchain with **Vite 5** for the dev server and
  production build.
- Replaced Jest + react-scripts test runner with **Vitest 2** (jsdom environment,
  v8 coverage, html/lcov/json-summary reporters).
- Migrated the entry point from `src/index.tsx` to **`src/main.tsx`** and the HTML

- Migrated the global test setup from `src/setupTests.ts` to
  **`src/test-setup.ts`**, and updated `vite.config.ts` to point at the new file.
- Kept a deliberate `.gitkeep` in `public/` to preserve the directory after the old
  `public/index.html` stub is removed.

The CRA migration left three tracked files in the tree that are no longer used by
**any** code path, build step, configuration, or test. They are inert, empty, and
unreferenced. They exist solely as historical artifacts of the CRA scaffold.

| Path | Size | Blob SHA | Referenced by |

| `public/index.html` | 0 bytes | `e69de29bb2d1d6434b8b29ae775ad8c2e48c5391` | ❌ Nothing — Vite uses the root [`index.html`](./index.html) |
| `src/index.tsx` | 0 bytes | `e69de29bb2d1d6434b8b29ae775ad8c2e48c5391` | ❌ Nothing — Vite uses [`src/main.tsx`](./src/main.tsx) |
| `src/setupTests.ts` | 0 bytes | `e69de29bb2d1d6434b8b29ae775ad8c2e48c5391` | ❌ Nothing — Vitest uses [`src/test-setup.ts`](./src/test-setup.ts) |

### Why they are 100% safe to remove

- **All three blobs are the same empty-file SHA** (`e69de29b…`). Removing them
  removes *zero bytes* of content.
- The root `index.html` loads `/src/main.tsx` directly — it does **not** reference
  `src/index.tsx` or `public/index.html`.
- `vite.config.ts` line 25 reads `setupFiles: ['./src/test-setup.ts']` — it does
  **not** reference `src/setupTests.ts`.
- `package.json` does not import or list any of the three paths.
- No source file, test file, hook, or fixture imports any of them.
- `tsconfig.json` and `tsconfig.node.json` do not enumerate inputs by name — the
  TypeScript program graph depends on actual imports, which do not exist.
- The `public/` directory is preserved by an intentional
  [`.gitkeep`](./public/.gitkeep), so removing `public/index.html` will not delete
  the directory from the tree.

### The single one-line command

A human with local shell access against this repository can finish v0.6.0 with:

```bash
git rm public/index.html src/index.tsx src/setupTests.ts \
  && git commit -m "chore: remove dead CRA stub files (v0.6.0 cleanup)" \
  && git push
```

> No code, configuration, build, or test needs to change. No CI adjustment is
> required. No new commit will be needed from automation. This is the **last**
> commit in v0.6.0.

## 3. Why this commit was not made from automation

This repository is operated through the GitHub Contents API (file get / list /
create / edit). That API surface has **no file-deletion operation**:

- `create_file` returns HTTP `422 Unprocessable Entity` when the target path
  already exists.
- `edit_file` overwrites the blob, but the orphan files are already empty blobs,
  and the operation cannot remove the path from the Git tree.

Removing a file from a tree requires either:

1. A `git rm` invocation from a local clone (the command shown above), or
2. The lower-level Git Data API (trees + commits + refs), which is not exposed
   to the automation that produced this file.

For full background see [`docs/_orphan_marker.md`](./docs/_orphan_marker.md),
which intentionally remains in its pre-removal state so that the governance
documents continue to reflect the codebase honestly.
